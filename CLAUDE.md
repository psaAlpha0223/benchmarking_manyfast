# CLAUDE.md

이 파일은 Claude Code가 이 프로젝트를 개발할 때 반드시 따라야 하는 규칙이다.
모든 규칙은 개발 시작부터 종료까지 항상 적용된다.

---

## 1. 프로젝트 개요

내부 팀 간 AI 솔루션 요청의 모호함을 줄이기 위한 기획 자동화 웹 서비스.
요청자가 텍스트·기능 태그·파일을 입력하면 PRD, 기능명세서, 유저플로우, 와이어프레임을 자동 생성한다.

**PRD 문서:** `PRD.md` (항상 최신 버전 기준으로 개발)

---

## 2. 기술 스택 (고정 — 임의 변경 금지)

| 영역 | 기술 |
|------|------|
| 프론트엔드 | Next.js (App Router) |
| 백엔드 | FastAPI |
| AI | Anthropic Claude API (`claude-sonnet-4-6`) |
| 스타일링 | Tailwind CSS |
| DB | Supabase (PostgreSQL + Storage) |
| 인증 | Supabase Auth (이메일/비밀번호) |
| ORM | supabase-py |
| 기능명세서 트리뷰 | @xyflow/react |
| PRD 렌더링 | react-markdown + remark-gfm |
| 유저플로우 렌더링 | mermaid.js |

기술 스택을 변경해야 할 상황이 생기면 **즉시 중단하고 사용자에게 알린다.**

## 4. 핵심 개발 규칙

### 4.1 PRD 기준 개발

- 모든 개발은 PRD를 기준으로 한다.
- 개발 중 PRD와 다르게 구현해야 하는 상황이 생기면 **즉시 중단**하고 아래를 전달한다.
  - 어떤 부분이 PRD와 다른지
  - 왜 달라질 수밖에 없는지
  - 어떤 선택지가 있는지 (최소 2가지 제안)
- 사용자가 의사결정한 후에만 계속 진행한다.

### 4.2 에러 처리

- 에러가 발생하면 **즉시 중단**하고 아래를 전달한다.
  - 어느 파일 / 어느 라인에서 발생했는지
  - 에러 메시지 원문
  - 해결 방법 제안 (최소 2가지)
- 사용자가 해결 방법을 선택한 후에만 계속 진행한다.
- 에러를 임의로 우회하거나 무시하지 않는다.

### 4.3 단계별·모듈별 개발

- 기능 전체를 한 번에 구현하지 않는다.
- 아래 순서로 모듈 단위로 개발하고, 각 모듈 완료 후 사용자에게 보고한다.

```
Phase 1. 인증 (로그인 / 비밀번호 변경 / 재설정)
Phase 2. DB 마이그레이션 (users / requests / outputs 테이블)
Phase 3. Input 플로우 (Step 1 → Step 2 → Step 3)
Phase 4. Claude API 연동 (인터뷰 질문 생성 → Output 생성)
Phase 5. Output 렌더링 (PRD → 기능명세서 → 유저플로우 → 와이어프레임)
Phase 6. 요청 이력 저장 및 조회
```

- 각 Phase 시작 전 반드시 사용자에게 "Phase N 시작합니다. 진행할까요?" 라고 확인한다.
- 각 Phase 완료 후 "Phase N 완료. 다음으로 넘어갈까요?" 라고 보고한다.

### 4.4 PRD 업데이트

- 모든 Phase가 완료된 후, PRD와 달라진 부분이 있으면 아래 형식으로 정리해서 알린다.

```
[PRD 변경 사항 보고]
- 변경 항목: (섹션명)
- 기존 내용: ...
- 변경된 내용: ...
- 변경 이유: ...
```

- 사용자가 승인하면 PRD를 새 버전으로 업데이트한다.
- 버전 형식: `v{major}.{minor}` (예: v1.4 → v1.5)

---

## 5. 코드 작성 규칙

- 환경변수는 반드시 `.env` / `.env.local`에서 읽는다. 코드에 직접 입력하지 않는다.
- 새 파일 생성 시 PRD 폴더 구조를 따른다. 임의로 구조를 바꾸지 않는다.
- DB 스키마 변경이 필요하면 반드시 `migrations/` 에 SQL 파일로 추가한다.
- 커밋 메시지는 한국어로 작성한다. 형식: `[Phase N] 작업 내용`

---

## 6. 금지 사항

- PRD에 없는 기능을 임의로 추가하지 않는다.
- 기술 스택을 임의로 변경하지 않는다.
- 에러를 무시하거나 `try/catch`로 덮어씌우지 않는다.
- 사용자 확인 없이 다음 Phase로 넘어가지 않는다.
- `.env` 파일을 Git에 커밋하지 않는다. (`.gitignore`에 반드시 포함)
